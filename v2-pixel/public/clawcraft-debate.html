ROUND 1

后端方案选择：插件 vs 独立服务

🔬 Ottor 提问

A) OpenClaw 插件（in-process） vs B) 独立 Express 服务器（轮询/外部）。已确认的插件 API（源码实证）：hooks、onAgentEvent、registerHttpRoute、registerService。

🤖 GPT-5.4 回复

选 A（OpenClaw 插件） — ClawCraft 是"活体鱼缸"，核心需求是低延迟事件流 + 单一真相状态。

4 层数据流架构：

📡 采集层(hooks+onAgentEvent) → 🗃️ 聚合层(registerService+Map) → 🌐 API层(HTTP+SSE) → 🖥️ 前端消费

风险识别：插件崩溃影响 Gateway、thinking 事件量大需采样、事件乱序需幂等、Gateway 重启丢状态、CORS/反代 SSE 兼容

结论："A 是事件原生架构，B 是外部观察架构；做数字鱼缸，应选 A。"

关键决策

  • 原生 PixiJS + SceneManager（不用 @pixi/react）
  • React absolute overlay 悬浮在 canvas 上
  • 一张大等距背景图 + 固定建筑槽位 JSON
  • 探险者 预设路径点 + Bezier 插值
  • spritesheet + AnimatedSprite：idle/walk/cast/error/return
  • 渲染 60fps，逻辑更新 10-15Hz
  • 素材：开源像素包 (Kenney/CraftPix/itch.io CC0)

MVP 素材：1 底图 · 2-3 Agent Hall · 4 类建筑 · 1 套探险者 atlas(5 动作) · 2 套特效 · HUD 图标

ROUND 4

插件代码骨架

export default {
  id: 'clawcraft',
  register(api) {
    sm = new StateManager(api.logger);
    api.registerService({ id: 'clawcraft-state', start, stop });
    
    api.on('session_start', (e, c) => sm.upsertSession(...));
    api.on('session_end',   (e, c) => sm.endSession(...));
    api.on('before_tool_call', (e, c) => sm.startTool(...));
    api.on('after_tool_call',  (e, c) => sm.finishTool(...));
    api.on('subagent_spawned', (e, c) => sm.linkSubagent(...));
    api.on('subagent_ended',   (e, c) => sm.unlinkSubagent(...));
    
    api.registerHttpRoute({ path: '/clawcraft/health', auth: 'gateway', ... });
    api.registerHttpRoute({ path: '/clawcraft/state',  auth: 'gateway', ... });
    api.registerHttpRoute({ path: '/clawcraft/events', auth: 'gateway', ... });
  },
  activate(api) {
    api.runtime.events.onAgentEvent((ev) => sm.ingestRuntime(ev));
  }
}

StateManager： snapshot() · upsert/end Session · start/finish Tool · link/unlink Subagent · subscribe()

SSE： text/event-stream · 首包 snapshot · 后续 patch · 15s 心跳 · 5s 重连

3 端点： /clawcraft/health · /clawcraft/state · /clawcraft/events

ROUND 5

Zustand Store + SSE→PixiJS 数据流

核心原则：SSE 不绕过 Zustand。 Zustand 做"事实状态+动画意图"单一真源。 
type Store = {
  entities:  { sessions: Record<id, SessionVM>, agents: Record<id, AgentVM> },
  animation: { queues: Record<sid, AnimIntent[]>, active: Record<sid, AnimIntent|null> },
  playback:  { speed: 1|2|4, paused: boolean },
  ui:        { selectedSessionId?: string },
  ingest(e: SSEEvent): void,
  ack(sessionId, intentId): void,
}
SSE event → store.ingest() → applyDomainEvent + mapEventToIntent → per-session FIFO → PixiJS play → ack() → next

动画队列：per-session FIFO，同时只播 1 个主动画，不同 session 可并行

时间缩放：speed 只影响 Pixi ticker dt，不改变状态提交顺序

ROUND 8 — DEVIL'S ADVOCATE

🔥 魔鬼辩护

⚔️ 攻击 1 — 插件耦合：OpenClaw 升级可能 break 全部事件接口，in-process 崩溃拖主进程。
🛡️ 反驳：可用薄桥接缓解；Plugin SDK 已有正式类型定义，breaking change 概率低。
⚔️ 攻击 2 — PixiJS 过重：30 sprite 上 WebGL 杀鸡用牛刀。
🛡️ 反驳：MVP 确实不值；100+ 单位后才体现价值。可先 CSS 方案验证。
⚔️ 攻击 3 — AoE 风格≠用户需求：用户要秒读状态，不是看鱼缸。
🛡️ 反驳：若目标是品牌展示/陪伴感可成立；若是工作台则方向错。定位为"第二屏"。
⚔️ 攻击 4 — SSE 在代理/CDN 下不稳：轮询更可靠。
🛡️ 反驳：内网直连 SSE 丝滑；可默认 SSE + 轮询 fallback。
⚔️ 攻击 5 — ROI 存疑：8-12 天做观赏层，谁是目标用户？
🛡️ 反驳：作为 demo/传播资产，品牌差异化；不是生产工具，是体验层。

替代方案对比

方案优势劣势工时
A: 插件+PixiJS+SSE ✅实时、表现力强、贴 OpenClawPixi 门槛、插件耦合7-12d
B: Express+轮询+CSS最快落地、易调试延迟高、动画上限低4-7d

ROUND 10 — FINAL

🏆 最终架构决策

一句话定位

ClawCraft 是把 OpenClaw 运行态做成可挂机、可一眼读懂的等距 RTS 数字鱼缸。

5 条关键决策

  1. 数字鱼缸优先 — MVP 只读，先证明观赏+监控价值
  2. 空间语义 — Agent=领主大厅+英雄 · Session=探险者 · Skill=分类建筑
  3. 插件直连 — Gateway 内 hook + onAgentEvent，低延迟单一真相
  4. 双层前端 — React 管 HUD，PixiJS 管世界
  5. 原地施法 — 短任务不寻路，建筑远程响应，避开大坑

数据流全景

Gateway hooks → StateManager → /state + SSE → Zustand → AnimIntent FIFO → PixiJS render → React HUD overlay

技术栈

TypeScript Plugin · Vite · React · PixiJS · Zustand · Tailwind · shadcn/ui

MVP 交付物

固定等距地图 · 2-3 Agent 大厅 · Session 生成/施法/结束 · 状态可视化 · XP/人口/成本 HUD · 只读详情 · 断线重连

路线图

P1 假数据地图 → P2 接真实事件 → P3 最小动效+HUD → P4 打磨部署

最大风险

事件语义→动画映射失真。缓解：有限状态机、最小动画字典、per-session FIFO、先准后美。

成功标准

  • ✅ @dora 愿意长开当第二屏
  • ✅ 3 秒内看懂"谁在干什么/是否异常"
  • ✅ 稳定不卡顿
